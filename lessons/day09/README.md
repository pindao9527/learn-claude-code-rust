# Day 09: 智能体团队 (Agent Teams)

> *"老板动嘴, 小弟跑腿, 信箱传情"* —— 实现基于文件锁的持久化多 Agent 协作架构。

## 1. 目标

在之前的章节中，我们实现了子智能体（Subagent）和后台任务（Background Tasks）。今天我们将迈出最关键的一步：**从“单兵作战”转向“兵团作战”**。
我们要构建一个持久化的 Agent 团队，每个成员都有自己的 ID、角色和独立的信箱。

## 2. 团队通讯架构

不同于内存中的变量传递，团队模式采用“异步信箱”机制，这让 Agent 能够像人类团队一样异步沟通。

```text
      Lead Agent (主管)              Teammate Agent (成员)
    +------------------+           +------------------+
    | agent_loop       |           | _teammate_loop   |
    |                  |           |                  |
    | [Spawn Tool] ----|---------->| [Tokio Spawn]    |
    |                  |           |                  |
    | [Send Msg] ------|---------->| [Inbox.jsonl]    |
    |       ^          |           |        |         |
    |       |          |           | [Read & Clear]   |
    | [Read Inbox] <---|-----------| [Send Reply]     |
    +------------------+           +------------------+
```

## 3. Rust 中的实现挑战

虽然 Python 版通过简单的 `append` 写入文件，但在 Rust 高并发环境下，如果不加保护，会导致以下灾难：
1. **数据错乱**：多个 Agent 同时写入同一个 JSONL 文件，导致行内容交织，变成无法解析的废纸。
2. **状态丢失**：在“读取并清空”信箱时，如果读取和删除不是原子的，会导致消息丢失。

为了解决这些问题，我们引入了 **`fs2` 跨平台文件锁**。通过 `lock_exclusive()`，我们确保了每一行消息的写入和每一次信箱的清空都是“绝对安全”的。

## 4. 今日任务

1. **MessageBus**: 封装带 `fs2` 独占锁的 JSONL 读写逻辑。
2. **TeammateManager**: 利用 `Arc<Mutex<TeamConfig>>` 跨线程安全地维护团队花名册。
3. **_teammate_loop**: 编写后台 Agent 的死循环，使其具备自主思考与回复能力。
4. **大脑升级**: 改造 `main` 循环为嵌套结构，解决 LLM 在多步工具调用时的“慢半拍”响应问题。

> **架构注记**：我们为什么不直接用 `mpsc` 频道通讯？因为文件信箱提供了**持久化**的可能。即使程序崩溃重启，未读的指令依然静静地躺在 `.team/inbox` 里，这才是生产级 Agent 系统该有的韧性。
